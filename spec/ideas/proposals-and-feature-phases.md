# Idea: Proposals and Feature Phases

**Owner:** alex

## Problem Statement

How might we make the proposal primitive (the SpecScore concept that captures "change a Feature after it has shipped") discoverable under its more common name "change request", codify the multi-phase Feature delivery pattern that uses proposals as the mechanism, and surface a Feature's pending and approved proposals directly in the Feature README so readers see planned phases at a glance?

## Context

The proposal primitive is forward-referenced from two existing canonical Features — `spec/features/feature/README.md` describes proposals as "change requests attached to a feature" living at `{feature}/proposals/<slug>.md` with lifecycle `draft → submitted → approved → implemented`, and `spec/features/plan/README.md` accepts `change-request` as a valid `Source type` value with the proposal as the trigger object. But the canonical `proposal` Feature itself does not yet exist at `spec/features/proposal/` or `spec/features/proposals/`. The directory `spec/features/proposals/` is not present anywhere in the spec tree, so newcomers reading either of the forward references hit a dead link when they click through.

Terminology drifts across the same concept: prose mentions "proposal", the SpecScore.Studio toolbar's URL grammar exposes the action as `?op=request-change` and labels the link "Request change" (per `spec/ideas/studio-toolbar.md` and the rendered Feature READMEs), the `plan` Feature uses the value `change-request` in its `Source type` field, and informal team talk uses "change request". A newcomer cannot easily tell that all four labels point at the same object.

The pattern of phased Feature delivery surfaced concretely in `spec/ideas/journal-and-summary.md`: Phase 1 ships as a Feature (defines the new capability), Phase 2 ships as a proposal attached to that Feature (mutates the capability once a prerequisite Feature lands). The pattern is reusable across SpecScore — any time a capability has a "ships now" core and a "lights up later" extension blocked on a separate Feature, the same shape applies — but the convention lives nowhere canonical. A reader of a Feature README also has no way to see "what change requests are in flight against this Feature?" without listing the `proposals/` directory by hand.

## Recommended Direction

Land three coordinated changes in one release:

**(1) Specify the canonical `proposal` Feature.** Create `spec/features/proposal/README.md` (singular, matching `feature`, `task`, `idea`) that defines: the path layout (`{feature}/proposals/<slug>.md`), the title prefix and body-metadata schema, the lifecycle (`draft → submitted → approved → implemented`), the relationship to plans (a proposal can be the `Source` of a plan with `Source type: change-request`), and an optional structured `**Phase:**` metadata field (positive integer) for proposals that constitute ordered phases of a multi-phase delivery. Update the dangling forward references in `feature` and `plan` to point at the real Feature.

**(2) Establish "change request" as the canonical user-facing alias for "proposal".** The directory name `proposals/` stays — filesystem churn for a terminology win is bad value. Instead, every prose mention introduces "change request (proposal)" on first use per document; CLI command names land as `specscore proposal new` (canonical, matches filesystem and other resource commands) with `specscore change-request new` as a documented alias; the studio toolbar's `?op=request-change` verb is documented as the user-facing entry point that creates a proposal artifact under the hood. The lint rule that validates titles accepts both `# Proposal: <Title>` and `# Change Request: <Title>` as equivalent dispatch keys, with `# Proposal: <Title>` as the canonical form a scaffolder emits.

**(3) Codify the phased-delivery pattern as a documented convention in the `feature` Feature, and surface in-flight proposals on every Feature README.** Add prose to `feature/README.md` explaining: "If a Feature's implementation is naturally split into ordered phases where later phases depend on prerequisites not yet ready, Phase 1 ships as the Feature itself (defining the new capability) and subsequent phases ship as proposals (change requests) attached to it. Each phased proposal SHOULD carry a `**Phase: N**` metadata field." Add a new `## Proposals` (or `## Change Requests` — Feature-spec time decision) index section to the Feature README convention, generated and validated by lint from filesystem state at `{feature}/proposals/`, listing each proposal with status, optional phase number, and title. The section is auto-omitted when the directory is empty; lint never forces a "Proposals: none" placeholder on Features that have never had one.

The three changes are coordinated because they reinforce one another: the `proposal` Feature defines the primitive, the `feature` Feature's prose explains when to reach for it (phased delivery) and surfaces it (proposals index), and the terminology alias makes the whole thing discoverable to readers who arrive with the term "change request" instead of "proposal".

## Alternatives Considered

1. **Rename the directory `proposals/` → `change-requests/`.** Lost because "proposal" is wired through `feature/README.md` and `plan/README.md` and the value `change-request` already exists as the `Source type` enum on plans; churning a filesystem convention to win a terminology argument creates more pain than it solves. The alias-in-prose approach gets the discoverability win without the rename cost.
2. **Drop the proposal concept entirely and use sibling Features for every phase.** Lost because a Phase-2-style change is structurally a *mutation* of an existing capability, not a new capability — and `plan/README.md` already draws that distinction explicitly: "A feature spec defines something new. A change request (implemented as a proposal) mutates something that already exists." Sibling Features double up the spec surface and weaken the spec-vs-mutation distinction the model already has.
3. **Make phases a first-class metadata concept on Features without going through proposals.** Lost because phases are exactly when a Feature gets mutated over time; the proposal lifecycle (`draft → submitted → approved → implemented`) already encodes the state machine a phase needs. Inventing a parallel `Phase Status` concept duplicates state and forces a second lifecycle to validate.
4. **Auto-scaffold placeholder proposals for planned future phases as IOUs** (e.g., a `Phase 2 — TBD` stub created at the same time as the Phase 1 Feature). Lost because proposals should be drafted when their acceptance criteria are knowable, not stubbed as forward declarations; a stub proposal with no AC pollutes the proposals index and lies about what is actually planned vs merely intended.
5. **Make the `## Proposals` index section mandatory on every Feature, including those with zero proposals (rendered as "None at this time").** Lost because most Features will have zero in-flight proposals most of the time; forcing the empty section onto every README is noise. Lint can deterministically generate the section only when content exists, which is the same pattern used by other index sections like Source References and Plans.
6. **Specify the canonical `proposal` Feature in this Idea but defer the phased-delivery convention and the proposals index to a follow-on.** Lost because the phased-delivery convention is the actual user-facing reason proposals matter for many readers; without it, the `proposal` Feature lands as a primitive with no explained use case. Bundling all three changes keeps the why and the how in the same release.

## MVP Scope

A focused release that ships three coordinated artifacts in one cycle: (1) the canonical `proposal` Feature at `spec/features/proposal/README.md` with path layout, body-metadata schema, the `draft → submitted → approved → implemented` lifecycle, the relationship to plans, and the optional `**Phase: N**` field — fully replacing the current forward references with a real Feature; (2) a change request (proposal) against the existing `feature` Feature that adds the phased-delivery convention prose and the `## Proposals` index section to the Feature README schema, plus a lint rule that auto-generates the section from `{feature}/proposals/` filesystem state when the directory is non-empty; (3) a small terminology pass across `feature/README.md`, `plan/README.md`, and CLI help text introducing "change request (proposal)" on first use, plus the `specscore change-request` CLI alias for `specscore proposal`. The release ships no studio UI changes (the toolbar verb `?op=request-change` is already locked), no rename of the `proposals/` directory, and no backfill of `## Proposals` sections on Features that have never had a proposal — the section appears automatically the moment a first proposal lands. If a reader of a Feature README can see at a glance "what change requests are open against this capability?" and a writer reaching for "I want to change this Feature" can find the `proposal` Feature in one search, the goal is met.

## Not Doing (and Why)

- Renaming the `proposals/` directory to `change-requests/` — filesystem churn across every Feature that has ever had a proposal, for a terminology win that the prose alias already delivers
- Inventing a separate first-class `Phase` artifact type outside the proposal lifecycle — the proposal state machine (`draft → submitted → approved → implemented`) already covers what a phase needs; a parallel concept duplicates state
- Auto-scaffolding placeholder proposals for "future planned phases" as IOUs — proposals should be drafted when their acceptance criteria are knowable, not stubbed as forward declarations that pollute the index without saying anything testable
- Cross-Feature proposals (a single proposal that mutates multiple Features simultaneously) — out of scope; if the use case becomes real it deserves a different artifact type, not a hack on top of single-Feature proposals
- Forcing the `## Proposals` index section onto Features that have zero proposals — lint emits the section only when filesystem state has content, matching the pattern used by other auto-generated index sections (Source References, Plans)
- Backfilling Phase metadata on existing Features that did not ship in phases — the convention is forward-looking; existing Features adopt the section the moment their first proposal lands
- Changing the studio toolbar verb spelling from `?op=request-change` — already in the locked toolbar contract per the studio-toolbar Idea; this Idea documents the verb as the canonical user-facing entry point but does not touch its wire format
- Adding "change request" as a second filesystem name alongside `proposals/` (e.g., scanning both `{feature}/proposals/` and `{feature}/change-requests/`) — one canonical filesystem layout; aliases live in prose and CLI surface, not on disk
- Auto-deleting or archiving proposal files once their `implemented` content lands in the Feature README — proposals are kept as historical record, similar to archived ideas; deciding the exact retention shape (keep-in-place vs move to `proposals/implemented/`) is deferred to Feature spec
- Specifying the canonical `proposal` Feature without bundling the phased-delivery convention and the proposals index — the three changes reinforce one another; splitting them ships the primitive without the explained use case

## Key Assumptions to Validate

| Tier | Assumption | How to validate |
|------|------------|-----------------|
| Must-be-true | The proposal lifecycle as forward-referenced from `feature/README.md` and `plan/README.md` (`draft → submitted → approved → implemented`) is the right state machine and needs no fundamental rework when the canonical `proposal` Feature is finally specified. Source files: `spec/features/feature/README.md`, `spec/features/plan/README.md`. | Audit the two forward references end-to-end; enumerate every operational question the lifecycle leaves open (who advances `submitted → approved`? what gates a transition? does `implemented` require the Feature README to actually contain the proposal's content?); resolve gaps inside the canonical Feature spec. If the gaps are deep enough to need a different state machine, this Idea expands scope or splits. |
| Must-be-true | Lint can deterministically generate and validate the `## Proposals` index section in a Feature README from filesystem state at `{feature}/proposals/`, using the same machinery that generates other index sections today (Source References, Plans). | Sketch the lint rule by analogy to existing index generators; confirm no new directives are needed; verify the rule can be added under the existing `feature` Feature contract without changing other lint rules. |
| Must-be-true | Introducing the CLI alias `specscore change-request <subcommand>` for `specscore proposal <subcommand>` is implementable in the existing CLI framework without creating two separate code paths or doubling test surface. | Inspect the existing CLI subcommand registration; confirm the framework supports command aliases as a first-class concept (most cobra-style CLIs do). If aliases require duplicate registration, evaluate whether the alias is worth the maintenance cost or whether prose-only aliasing is enough. |
| Should-be-true | The "change request (proposal)" prose alias pattern does not confuse readers familiar with either term in isolation — i.e., introducing the alias parenthetically reads as helpful clarification rather than as ambiguity about which is canonical. | Read sample copy aloud to 2–3 SpecScore-using developers; check for conflation with adjacent concepts like GitHub PR review comments, RFC documents, or design-doc proposals. If the alias creates more confusion than it removes, fall back to "proposal" as the sole term and document "change request" only in a single glossary entry. |
| Should-be-true | Most Features will have zero or one in-flight proposals at any given time, so the auto-generated `## Proposals` index section is a low-noise addition to Feature READMEs (empty or near-empty most of the time, with the section auto-omitted when empty). | Spot-check existing Features and project forward; assume a steady-state rate of ~1 proposal per Feature per quarter; verify the section does not become a wall of text on heavily-iterated Features. If projected counts are high, define a "recent / archived" split in the index. |
| Should-be-true | The optional `**Phase: N**` metadata field is the right primitive for phased delivery — i.e., a positive-integer sequence number per Feature is sufficient, and no richer phase-graph (DAG of dependencies between phases) is needed in v1. | Confirm with the journal-and-summary Idea (the worked example): does Phase 2 depend on anything other than "a foundational layered-config Feature shipped"? If real phases turn out to need a graph of prerequisites, the field becomes structured (e.g., `**Phase:**` block with `number` and `depends_on`); if they remain a linear sequence, a bare integer suffices. |
| Might-be-true | Future SpecScore.Studio tooling will consume the `**Phase: N**` field to render timeline-style views of a Feature's phased delivery (Phase 1 implemented, Phase 2 in review, Phase 3 draft, etc.). | Do not validate up-front; ship the optional field and let studio decide whether to consume it. The CLI and lint contract is the same either way. |
| Might-be-true | Future tooling will benefit from a back-reference in the Feature README's frontmatter listing all proposal slugs (a `**Proposals:**` managed-state line, similar to `**Promotes To:**` on Ideas) in addition to the prose `## Proposals` index section. | Defer; the prose index section is sufficient for v1. If a structured frontmatter back-reference becomes useful (for example, for cross-Feature queries by Synchestra), add it as a follow-on. |

## SpecScore Integration

- **New Features this would create:** The canonical `proposal` Feature at `spec/features/proposal/README.md` covering path layout, body-metadata schema, lifecycle, plan-source semantics, and the optional `**Phase: N**` field. Feature specifying the proposals index section + phased-delivery convention may land as either a sibling Feature or as a change request (proposal) against the existing `feature` Feature — the latter is more honest because it is structurally a mutation of `feature`, but doing so before the canonical `proposal` Feature exists creates a chicken-and-egg ordering problem. The likely resolution: the canonical `proposal` Feature ships first in this release, and the changes to the `feature` Feature ship as a proposal against `feature` immediately after, in the same release, as the dogfooded first concrete use of the primitive.
- **Existing Features affected:** `feature` — gains the phased-delivery prose, the `## Proposals` index section as a documented Feature README convention, and updates the forward reference to the now-real `proposal` Feature; `plan` — updates the forward reference to the now-real `proposal` Feature; the `Source type: change-request` enum value stays unchanged; `studio-toolbar` — no functional change, but the `?op=request-change` verb is documented as the canonical user-facing entry point that creates a proposal artifact; `repo-config` — no change (proposals do not introduce new `specscore.yaml` config); `idea` and `task` — no change.
- **Dependencies:** None hard-blocking; this Idea formalizes a primitive already forward-referenced from existing Features. Sequencing-wise it ideally lands before any sibling Idea creates its first concrete proposal — notably `journal-and-summary`, whose Phase 2 is the first planned concrete use. If `journal-and-summary` Phase 1 ships before this Idea promotes, the Phase 2 proposal can still be authored against the forward-referenced lifecycle, but it sits in a slightly under-defined state until this Idea lands.

## Outstanding Questions

- Title prefix for a proposal artifact: `# Proposal: <Title>` (matches the filesystem directory `proposals/` and the canonical CLI command `specscore proposal new`) or `# Change Request: <Title>` (matches the user-facing alias and the studio toolbar verb `?op=request-change`)? Schema dispatch keys are conventionally tied to filesystem; lean is `# Proposal: <Title>` with all *prose* using "change request (proposal)". The trade-off deserves a deliberate decision at Feature spec time and may affect lint rule design.
- Section heading on Feature READMEs: `## Proposals` (matches filesystem, terse) or `## Change Requests` (matches the user-facing alias, more discoverable)? Lean `## Proposals` for filesystem consistency, with the auto-generated section's first sentence reading "Open change requests (proposals) against this feature." to bridge the terminology.
- Structure of the `**Phase:**` metadata field on a proposal: bare positive integer (`**Phase:** 2`) for a linear sequence, or a structured block (`**Phase:**` with sub-fields `number`, `depends_on`) for a phase graph? Linear is simpler; graph is more expressive. Lean linear; expand if the journal-and-summary worked example or another real case demands the graph.
- Ordering inside the auto-generated `## Proposals` index: by status (drafts at bottom, approved at top)? By phase number when present? By creation date? Lean reverse chronological by creation date with a secondary sort by phase number, which keeps the most-recent activity visible while preserving phase ordering for multi-phase Features.
- Retention model for `implemented` proposals: keep the proposal file in place at `{feature}/proposals/<slug>.md` indefinitely (matching the archived-Ideas convention), or move it to `{feature}/proposals/implemented/<slug>.md` to keep the active index clean? Lean keep-in-place; lint can filter the active index to non-`implemented` proposals without moving files.
- CLI command name: `specscore proposal new` (canonical, matches the filesystem and the other singular resource commands `idea new`, `task new`) with `specscore change-request new` as a documented alias? Lean yes; defer the exact alias mechanism (cobra command alias vs separate command that delegates) to Feature spec.
