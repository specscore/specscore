# Idea: Journal and Summary

**Status:** Specifying
**Date:** 2026-05-19
**Owner:** alex
**Promotes To:** journal-and-summary
**Supersedes:** —
**Related Ideas:** —

> **Referenced by [Decision D-0002](../decisions/0002-recap-storage-surfacing-and-producer-gate-boundary.md) (Accepted 2026-06-02).** D-0002 makes this journal the **rollup spine** of the recap model: the session recap and month report are projections of the journal over a time window, and the `recap.completed` / `verify.completed` events carry `report_path` + per-AC verdict counts so portfolio rollups link into the detail without scanning recap filenames for the time dimension. D-0002 also notes this Idea must be promoted to a Feature (it is the spine); that promotion is now done — see [`spec/features/journal-and-summary/`](../features/journal-and-summary/README.md) (Phase 1; Phase 2 cross-repo `journal.repo` remains a gated proposal).

## Problem Statement

How might we give users a low-friction view of recent SpecScore activity and rollup summaries (day / week / month) without taxing their flow or biasing any single command's latency?

## Context

Today the only record of SpecScore activity is git history plus the Synchestra event stream that skills emit (idea.drafted, idea.approved, feature.approved, task.*, etc.). There is no human-readable view of "what changed in this spec tree recently?" and no rollup over a day, week, or month. As repos accumulate dozens of features and hundreds of events, scanning git log becomes infeasible and the event stream is structured but not curated. Users want the answer to two questions surfaced cheaply: (a) what has been done recently, (b) what is the high-level summary for this day / week / month.

## Recommended Direction

Treat the journal as an append-only, low-cost write that happens at the existing Synchestra event-emission points (skills and CLI commands that already emit events also append one line per event). Treat aggregation as a pure on-demand projection — `specscore journal summary --period=day|week|month` reads the journal and synthesizes the rollup at query time, with no pre-computation, no scheduler, no commit hook, and no session-start cost. This sidesteps the fairness concern ("who pays for aggregation?") by making the answer "the person who asks for it, at the moment they ask."

By default the journal lives under a new top-level `_specscore/journal/` directory of the current repo (visible, not hidden, conventional underscore prefix for generated content — see Alternatives for why not `$specscore/`). The directory location is configurable via the `journal.repo` key: when set, journal writes (and reads) go to `<journal.repo>/_specscore/journal/` instead of the current repo. This enables a single shared journal across many source repos — e.g., every SpecScore-managed project on a developer's machine writes into `~/work-log/_specscore/journal/`, producing one unified activity stream.

Because committing a user-specific path like `~/work-log` into a shared `specscore.yaml` is hostile to teammates, the cross-repo `journal.repo` capability **depends on a separate foundational Feature** that introduces three-layer configuration resolution (`specscore.local.yaml` → `specscore.yaml` → `~/.specscore.yaml`, most-specific wins) across all of `specscore.yaml`. That foundational Feature is sequenced before this Idea's full capability ships, and this Idea does **not** carry layered config as a narrow journal-only precedent. As a result the work splits into two phases:

- **Phase 1** (ships first, no prerequisite Feature). Single-repo journal: writes always go to the current repo's resolved `journal.path` (default `_specscore/journal/`). Config lives in project-level `specscore.yaml` only. The `journal.repo` key is documented but rejected by the loader with a clear error pointing at the foundational Feature. Phase 1 delivers `journal show`, `journal summary`, deterministic auto-write, and explicit `journal write` — the entire single-repo experience, immediately useful.
- **Phase 2** (ships as a *proposal / change request* attached to the Phase 1 Feature, advanced after the foundational layered-config Feature lands). `journal.repo` activates. Users can point their journal at a personal `~/work-log` repo and aggregate across all SpecScore-managed projects on the machine. No other surface area changes — Phase 2 is a config-loader lift-up, not a journal redesign, which is precisely why it fits the proposal model (mutating an existing capability) rather than a sibling Feature.

Resolved config keys (Phase 1): `enabled` (default `true`), `path` (default `_specscore/journal/`) — fully overridable so users who dislike the `_specscore/` convention can point it anywhere (e.g., `path: journal/`, `path: .journal/`, `path: data/activity/`), optional `aggregation_periods` allowlist (default `[day, week, month]`). Phase 2 adds `repo` (default unset → use current repo). Setting `journal: null` or `journal.enabled: false` disables writes entirely; existing skills become no-ops on the journal call.

The on-disk format is **one event per file**, sharded by date, stored as an [inGitDB](https://ingitdb.com/) collection: `events/YYYY/MM/DD/<ISO8601-timestamp>-<machine-id>-<short-uuid>.json`. SpecScore CLI links against the [ingitdb-cli](https://github.com/ingitdb/ingitdb-cli) Go library (`pkg/dalgo2ingitdb`) and uses it as the **only** writer of journal records. A canonical `.ingitdb.yaml` schema at the journal-repo root declares the `events` collection and the event record shape, so reads work uniformly whether the user runs `specscore journal show` or `ingitdb select --from=events --where='type==idea.approved'` directly.

**Event record schema (flat fields, ingitdb-friendly for `--where` filters):**

- `type` — event type (e.g., `idea.drafted`, `idea.approved`, `feature.new`, etc.)
- `timestamp` — ISO 8601 UTC; the filename also encodes this for cheap directory pruning
- `machine_id` — short stable identifier of the writing machine (so multi-machine sync produces non-colliding filenames)
- `source_repo` — local filesystem path of the source repo root at write time (`git rev-parse --show-toplevel`)
- `source_origin` — verbatim git origin URL (`git remote get-url origin`), or omitted when no origin remote is configured
- `stream` — always populated per the resolution rules below
- `data` — type-specific payload (the structured event details — slug, status transition, etc.)

**Stream resolution at write time (precedence, first match wins):**

1. If `journal.stream: <value>` is explicitly set in the resolved config (Phase 2) → `stream = <value>`
2. Else if `journal.stream: null` is explicitly set (Phase 2) → `stream` is omitted from the event record (user opt-out)
3. Else if the source repo's git origin URL is parseable for an org (first path segment after the host, e.g., `git@github.com:specscore/specscore.git` → `specscore`) → `stream = <org>`, lowercased
4. Else (no origin remote, malformed URL) → `stream = <basename of source_repo>`, lowercased

Rules 3 and 4 fire in Phase 1 (no config key yet); rules 1 and 2 activate in Phase 2 once `journal.stream` is wired through the resolved config. Stream values are resolved at write time and persisted into the event record — moving a repo to a different org later does not retroactively re-tag historical events. The same applies to renaming a local directory: rule-4 events keep the original basename. Reads can then filter (`--stream=<name>`, `--no-stream`) or aggregate across all streams (the default), and `ingitdb select --from=events --where='stream==specscore'` works directly.

**Edge-case decisions baked into rule 3:**

- **GitLab subgroups** (`gitlab.com/group/subgroup/repo`) — first path segment only (`group`); subgroup granularity is achievable via explicit `journal.stream` in Phase 2.
- **Case normalization** — lowercase the org at write time so `SpecScore` and `specscore` never fragment a stream.
- **SSH host aliases** (`git@github-personal:org/repo.git` resolved via `~/.ssh/config`) — parse anyway; standard `host:org/repo` shape, alias resolution is below this layer's concern.

**Write paths** — there are exactly two, both funnelling through the SpecScore CLI as the single writer:

1. **Implicit (deterministic auto-write).** Every SpecScore CLI subcommand of the form `specscore <resource> new` or `specscore <resource> change-status` — across all current resources (`idea`, `task`, etc.) and all future resources added to the CLI — records the corresponding journal event as a side effect of its normal execution. The rule is structural, not enumerated: any new resource that introduces a `new` and/or `change-status` subcommand automatically inherits journal coverage at the moment it ships, with no additional wiring required. Skills that already invoke these commands get journal coverage "for free" with no extra calls. The auto-written event payload is derived deterministically from the invoking command's arguments and the resulting artifact state, so the same logical operation always produces the same event shape.
2. **Explicit.** Anything the CLI does not infer — skill-level checkpoints, custom workflow steps, ad-hoc notes from external automation — goes through `specscore journal write --type=<event-type> [--data='{...}']`. Skills never link against ingitdb directly; they always shell out to `specscore`. This keeps the dependency surface narrow: only the SpecScore CLI knows about ingitdb internals.

Because two machines never write the same file path (machine ID + uuid in the filename) and SpecScore is the only writer, cross-machine sync of a shared journal repo is a clean git fast-forward / pure-add merge with **no line-ordering or interleaving conflicts** to resolve. Date-sharded directories also let `journal summary --period=<p>` prune to the relevant date range without opening every event file. `specscore journal show --since=<duration>` lists recent events; `specscore journal summary --period=<p>` produces the rollup. Reads use the ingitdb library too (via the same `dalgo2ingitdb` adapter) for consistency with writes; both are pure operations against the journal repo's working tree.

## Alternatives Considered

1. **Scheduled / background aggregation** — a daemon or cron job pre-computes daily, weekly, monthly rollups and writes them as files. Lost because every triggering point has fairness problems: session-start delays the user's first prompt, commit hooks tax every `git commit`, and a separate daemon adds an operational dependency no other SpecScore capability has. On-demand projection costs the same compute total but charges only the person actually asking.
2. **Narrow journal-only precedent for layered config** — introduce three-layer resolution (`specscore.local.yaml` → `specscore.yaml` → `~/.specscore.yaml`) scoped to the `journal:` block only, as part of this Idea, so cross-repo `journal.repo` ships in one shot. Lost because the team chose to sequence a separate foundational layered-config Feature first; landing layering wholesale across all of `specscore.yaml` keeps the config loader simple (one code path) and prevents per-block layering rules from accumulating. This Idea instead splits into Phase 1 (project-only config, single-repo journal — ships immediately) and Phase 2 (cross-repo `journal.repo` activates once the foundational layered-config Feature lands).
3. **Project-only config forever (no cross-repo capability at all)** — never ship `journal.repo`; users who want cross-repo aggregation manually maintain symlinks or wrapper scripts. Lost because the cross-repo use case is one of the original requirements; deferring the *activation* of `journal.repo` to Phase 2 is acceptable, abandoning it is not.
4. **Pre-computed versioned rollup artifacts** (`_specscore/journal/summaries/2026-W20.md` committed to git) instead of on-demand summaries. Lost because the rollup is a deterministic projection of the journal — committing it duplicates state, invites drift, and creates merge conflicts on every cross-machine sync. Better to compute on demand; if users later want frozen weekly snapshots for release-note purposes, that is an additive `journal snapshot` Feature.
5. **`$specscore/` as the literal directory name** (the user's initial preference). Lost because `$` is a shell metacharacter — `cd $specscore` expands as a variable lookup, tab-completion behaves oddly, and many IDE search filters, file pickers, and config parsers choke on `$` in path segments. `_specscore/` preserves the visibility intent without the tooling hazard. The directory name remains fully configurable via `journal.path` for users who prefer a different convention.
6. **Single JSONL file** (`_specscore/journal/entries.jsonl`, append-only, one event per line). Lost because cross-machine sync of a shared journal repo produces unavoidable merge conflicts — git sees concurrent appends as overlapping line-ranges, and even when resolution is "merge both, sort by timestamp" the conflict markers must be handled per-merge. One-file-per-event sharded by date sidesteps this entirely: two machines never write the same path, so merges are pure adds.
7. **Convention-only inGitDB compatibility** (SpecScore writes one-event-per-file in an ingitdb-compatible layout but does not link against the ingitdb library or call the ingitdb binary; reads happen via filesystem walk). Considered to keep SpecScore's release cadence decoupled from a pre-v1 third-party tool. Lost because the team explicitly chose tight integration: SpecScore CLI links the ingitdb Go library and uses it as the single writer/reader. This buys schema validation, query semantics, and future feature parity (e.g., MCP server access to journal events) at the cost of a real build-time dependency. The risk of upstream API breakage is accepted and pinned in the Must-be-true assumptions below.
8. **Skills link against ingitdb directly** (each skill writes its own events). Lost because it spreads ingitdb knowledge across every consumer, multiplies the version-pin surface, and prevents the deterministic auto-write path. Funnelling all writes through `specscore` CLI keeps exactly one place that knows the journal schema and the ingitdb API.

## MVP Scope

**Phase 1** (1–2 week spike, ships immediately, no prerequisite Feature). Single-repo journal with project-only config: (1) the date-sharded one-event-per-file layout at `<journal.path>/events/YYYY/MM/DD/<ts>-<machine>-<uuid>.json`, written to the current repo at the resolved `journal.path`, with the ingitdb Go library (`pkg/dalgo2ingitdb`) as the underlying read/write engine inside the SpecScore CLI; (2) a canonical `.ingitdb.yaml` schema scaffolded at the journal-root on first write declaring the `events` collection and the full event record shape (`type`, `timestamp`, `machine_id`, `source_repo`, `source_origin`, `stream`, `data`); (3) write-time stream resolution implementing rules 3 and 4 of the precedence table (origin org via git remote parsing, else lowercased source repo basename) — every Phase 1 event has a populated `stream` field even without any user config; (4) the `journal:` block schema in project-level `specscore.yaml` only, with keys `enabled`, `path`, and `aggregation_periods` — `repo` and `stream` are documented but rejected by the loader with clear errors pointing at the foundational layered-config Feature and the Phase 2 follow-on; (5) deterministic auto-write of journal events from every `specscore <resource> new` and `specscore <resource> change-status` subcommand across all current and future resources — the rule is structural so any new resource that adds these subcommands inherits journal coverage automatically; (6) `specscore journal write --type=<event-type> [--data='{...}']` for the explicit write path used by skills and external automation; (7) two read commands — `specscore journal show --since=<duration> [--stream=<name>] [--no-stream]` and `specscore journal summary --period=day|week|month [--stream=<name>]` — both pure reads that use the same ingitdb library underneath and default to aggregating across all streams.

**Phase 2** (ships as a *proposal / change request* attached to the Phase 1 Feature, advanced after the foundational layered-config Feature lands). Two config keys activate together: `journal.repo` (point the journal at a different filesystem location for cross-repo aggregation) and `journal.stream` (override the inferred stream value per project — rules 1 and 2 of the precedence table; explicit value sets the stream, explicit `null` opts out). The loader stops rejecting both keys and resolves them through the three-layer config stack (`specscore.local.yaml` → `specscore.yaml` → `~/.specscore.yaml`). No other surface area changes; writes and reads continue to use the ingitdb library, the auto-write rule continues to cover every `new` / `change-status` command, and the file layout is unchanged. Phase 2 fits the proposal model (mutating an existing capability) rather than a sibling Feature.

The MVP intentionally ships no pre-aggregated artifacts, no versioned weekly summary files in git, no studio dashboard, no email or Slack digests, no journal-block-only layered-config precedent, no ingitdb integration in any consumer other than the SpecScore CLI itself. If the output is not embarrassing on day one (Phase 1: `specscore journal show --since=24h` in any SpecScore-managed repo prints recent activity from that repo, and `ingitdb select --from=events --where='type==idea.approved'` works against the same files for power users) the timing was right.

## Not Doing (and Why)

- Versioned rollup artifacts committed to git (`_specscore/journal/summaries/2026-W20.md`) — deferred; can be added as an additive `specscore journal snapshot` Feature once on-demand rollups prove their shape
- Pre-computed aggregation on session start, on commit, or via a background daemon — pre-computation is what creates the fairness concern; on-demand projection sidesteps it entirely
- Studio dashboard or PWA view of the journal — separate Feature owned by SpecScore.Studio; out of scope for the CLI capability
- Email or Slack digest integrations — that is a distribution channel; ship the canonical capability first, distribution Features can subscribe later
- Backfill of pre-journal history from git log or Synchestra event archives — the journal starts the day it is enabled; retroactive synthesis is a separate Feature with its own correctness questions
- Shelling out to the ingitdb CLI binary at runtime — SpecScore links the ingitdb Go library directly (`pkg/dalgo2ingitdb`) for both reads and writes; the user does not need the `ingitdb` binary installed for SpecScore's journal commands to work
- Spreading ingitdb dependency to skills, external automation, or third-party consumers — only the SpecScore CLI links against ingitdb; everything else writes via `specscore journal write` or via implicit auto-write from SpecScore subcommands
- Single JSONL file format for journal events — would force per-merge line-conflict resolution on cross-machine sync; one-file-per-event sharded by date eliminates the problem by construction
- Introducing layered config (`specscore.local.yaml`, `~/.specscore.yaml`) inside this Idea, even as a narrow journal-only precedent — the foundational layered-config Feature lands first across all of `specscore.yaml`; this Idea's Phase 2 consumes that Feature rather than building its own per-block layering rule
- Auto-scaffolding `specscore.local.yaml` into `.gitignore` — that belongs to the foundational layered-config Feature, not this Idea
- Network-mounted, symlinked, or remote-fetched journal targets — `journal.repo` resolves to a local filesystem path; team aggregation across machines is achieved by the user pushing the shared journal repo to a git remote, not by SpecScore reaching across the network at write time
- Automatic `git push` / `git pull` of the shared journal repo — sync cadence and remote setup are the user's responsibility; SpecScore only writes files into the working tree
- Per-event garbage collection, compaction, or retention enforcement — events are append-only forever in v1; if event-file counts become a git-performance problem, a later Feature can add bucketed compaction (e.g., monthly archive tarballs)
- Custom event filters beyond `--since` and `--period` (per-author, per-feature, per-event-type) — surface area kept small; can be added once real usage shows what filters matter
- Storing the journal directory as `$specscore/` (literal dollar prefix) — hostile to shells (`cd $specscore` expands as a variable), tab-completion, IDE search filters, and many config parsers; `_specscore/` preserves the user's visibility intent without breaking tooling
- Storing the journal inside `spec/` — `spec/` is human-curated content; the journal is tool-generated, so mixing them in one tree confuses the lint contract and the mental model
- Storing the journal under a hidden `.specscore/` directory — explicitly contradicts the user's stated preference for visibility of non-manual content

## SpecScore Integration

- **New Features this would create:** One Feature ("journal-and-summary") covering the inGitDB-backed event collection, the `.ingitdb.yaml` schema scaffolding, the project-level `journal:` config block, the auto-write rule over all `<resource> new` / `<resource> change-status` subcommands, the explicit `journal write` command, and the two read commands (`journal show`, `journal summary`). **Phase 2 (`journal.repo` activation) is a proposal — a change request — attached to the Phase 1 Feature**, not a sibling Feature, because it mutates an existing capability rather than defining a new one. It lives at `spec/features/journal-and-summary/proposals/<slug>.md` and follows the proposal lifecycle (`draft → submitted → approved → implemented`), sitting in draft/submitted state until the foundational layered-config Feature lands. A separate foundational Feature for `specscore.local.yaml` + `~/.specscore.yaml` across all of `specscore.yaml` is a hard prerequisite for the Phase 2 proposal to advance — owned outside this Idea.
- **Existing Features affected:** `repo-config` — adds the project-level `journal:` block schema; every existing and future SpecScore subcommand matching `<resource> new` or `<resource> change-status` — gains a deterministic auto-write side effect (`idea new`, `idea change-status`, future `feature new`, future `feature change-status`, future `task change-status`, etc.).
- **Dependencies:** (1) New build-time Go-module dependency on `github.com/ingitdb/ingitdb-cli` (specifically `pkg/dalgo2ingitdb` and the DALgo `dal.DB` adapter). This is the first external runtime dependency SpecScore takes on beyond the Go standard library and existing direct deps — flagged for review during Feature spec. (2) Phase 2 depends on a separate foundational layered-config Feature (`specscore.local.yaml`, `~/.specscore.yaml`, three-layer resolution across all blocks). Phase 1 has no such dependency and ships standalone.

## Open Questions

- **Stream topology — resolved.** Schema and resolution rules locked in (see Recommended Direction): events carry `source_repo`, `source_origin`, and `stream`. Stream is resolved at write time with four-rule precedence — explicit `journal.stream` config (Phase 2), explicit `journal.stream: null` opt-out (Phase 2), git origin org parsing, lowercased source-repo basename fallback. Hard separation across streams uses Phase 2's `journal.repo` (different journal repos per logical bucket); soft separation within one journal uses the `stream` field. Option (a) — multiple top-level collections in one journal repo — was dropped as dominated by (b)+(c). Edge cases (GitLab subgroups, case normalization, SSH host aliases, local-only repos) all decided per the bullets in Recommended Direction.
- Should `specscore journal write` accept an event from stdin (`--data-file=-`) as well as `--data='{...}'`, to make it ergonomic for long structured payloads in scripts? Likely yes, but defer to Feature spec.
